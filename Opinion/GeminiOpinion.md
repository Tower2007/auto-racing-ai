## 2026-05-06: 異常オッズ（センチネル）対策と「自己修正ループ」の機動力

直近の 3 コミット（`8dbac42` 〜 `3cc79f2`）による異常オッズ対策の変遷を確認した。これは「観測フェーズ」におけるシステムとチームの健全性を証明する、極めて質の高い一連の動きである。

### 1. 「実測値による前提の破壊」の速さ
- **評価**: `3cc79f2` において、自身が立てた「複勝最小 1.1」という前提を、わずか数分後に `bet_history.csv` の実データ（元返し 1.0 の実在）に基づき自己修正（OR → AND への変更）した点を高く評価する。
- **意義**: AI 運用で最も恐ろしいのは「誤った固定観念」によるデータの切り捨てだが、本プロジェクトでは「実際の財布（bet_history）が正しい」という真理が、エンジニアリング上のプライオリティ最上位に置かれていることが再確認された。

### 2. LEAD_MIN=5 短縮が生んだ「副作用」への即応
- **因果関係**: 発火タイミングを 5 分前に早めたことで、オッズ取得スクリプトが「システム上の過渡期（sentinel odds）」を拾う確率が上がった。この副作用を `EV=59.28` という誤発火から即座に検知し、`ODDS_MAX_CAP` 等のガードレールを設けた判断は極めて機敏である。
- **リトライ導線との整合性**: 異常値を単にエラーにするのではなく「NaN 化して後続のリトライ（60s x 2）に委ねる」設計は、`daily_predict.py` の再試行ロジックと見事に噛み合っている。

### 3. 今後の「共通化」への観点
- **重複の許容**: 現在、異常値処理ロジックが `daily_predict.py` と `streamlit_app.py` に重複している点について、「3 つ目のパターンが出てきて 3 回目の同期修正が必要になったら、その時点で utility 化を検討する材料にはなる」という判断は合理的である。現在の Phase A においては、リファクタリングによる未知のデグレードリスクを避けるべきであり、同期修正のコストが低い現状ではこのまま進むのが「運用の冷徹さ」に合致する。

### Gemini からの総括
この 16 分間での 3 コミットの連鎖は、本プロジェクトが**「データの歪みに気づき、実データで検証し、即座に整合性を取る」という自己修復能力**を十分に備えていることを示している。この「測定器の研ぎ澄まし」こそが、Phase A における最重要タスクである。引き続き、静観を継続する。

## 2026-05-06: プロジェクトの「成熟」と Phase A 運用への最終評価

直近数コミット（`6dd7ee0` 等）および Audit R9 までの「一連の動き」を俯瞰し、プロジェクトが単なる技術検証を超えて、極めて規律正しく成熟した運用フェーズに入ったことを確認した。

### 1. 5分前発火（LEAD_MIN=5）への移行という英断
- **意義**: `dynamic_scheduler.py` のリードタイムを 30 分から 5 分に短縮したことは、本プロジェクトにおける「実効期待値」を closing odds backtest の理論値に近づけるための、最も技術的に誠実な変更である。
- **リスクの最小化**: これにより、不確実な「期待値の先行指標」に頼るのではなく、市場が最も効率化した最終形に近いオッズでフィルタリング（thr=1.50）が可能になった。10 分前以前のオッズに存在していた「late money 前のノイズ」を排除できるメリットは、多少の投票時間不足リスクを上回る。

### 2. 構造的 Edge のドキュメント化と「4行ルール」の確立
- **戦略の言語化**: `docs/ev_strategy_findings.md` に「1番人気 overround 0.76」を明文化したことで、チーム全体（人間と全 AI）が「何を狩っているのか」を共通認識化できた。これは、的中率が落ちた際の「パニック的変更」を防ぐ強力な心理的アンカーになる。
- **規律の強制**: live n=100 までの「一切の変更禁止」という 4 行ルールは、AI 特有の「何かをいじって改善したい欲求（Recency Bias）」を封じ込める、極めて senior な設計判断である。

### 3. モニタリング・パイプラインの完成度
- **自動化の恩恵**: `fetch_order_history.py` による実損益の自動取得と、`weekly_status.py` への統合により、運用者の主観が入る余地を完全に排除した「真実のフィードバックループ」が完成している。
- **ヘルスチェックの深化**: WARN/NG をメール件名に反映させる等の R9 までの改善は、システムが「サイレントに死ぬ」ことを防ぐ実運用上の急所を押さえている。

### 自分が Claude / Codex なら気付かなかった視点
- **「LEAD_MIN=5 への依存」という新たな脆弱性**: 5 分前発火は理想的だが、autorace.jp 側の API レスポンス遅延や、ネットワーク遅延が発生した際の「猶予」を極端に削っている。現在の 60s x 2 retry が「間に合わなかったレース」の統計（締め切りによる未投票率）を、PnL とは別に追跡すべきである。利益が出ているように見えても、もし「期待値が高いレースに限って締め切りで買えない」という負の相関が隠れていた場合、実効 ROI は理論値からさらに下振れする。

### Gemini からの最終評価
現在の `auto-racing-ai` は、**「技術的野心」と「運用の冷徹さ」が高度にバランスした状態**にある。「いろいろ動きあった」の正体は、この**「運用規律のシステム化」**であると断定する。このまま n=100 まで一切のレバーを動かさず、静かにデータを蓄積し続けることを強く推奨する。

## 2026-05-06: MLロジック監査（Codex/Claude）の大局観レビュー

本日（5月6日）完了した ML ロジック監査により、モデルの健全性（リークなし）と、何よりも「期待値（edge）の源泉」が具体数値で特定されたことを高く評価する。

### 1. 「1番人気の overround 0.76」発見の戦略的意義
- **核心の特定**: Claude の追加検証（検証 6）で示された「1番人気だけが実力以上に売れている（期待値が極端に低い）」という構造的歪みは、本プロジェクトの存在意義を定義するものである。
- **ビジネスモデルの明確化**: 本システムは「優れた予測器」である以上に、「大衆の過剰な心理的バイアス（1番人気への安心感）を淡々と刈り取るフィルター」として機能していることが証明された。これにより、今後の改善の方向性が「予測精度の追求」から「バイアスの安定的な捕捉」へと、より投資効率の高い方へシフトできる。

### 2. 「市場効率化のジレンマ」への再警告
- **長期的リスク**: 特定された edge が「大衆のバイアス」に依存している以上、同様の AI 参入や、本システムの運用規模拡大（あるいは姉妹プロジェクトへの横展開）が、この 0.76 という歪みを 1.0 に近づけてしまう。
- **モニタリング項目への追加**: Claude が提案した「1番人気の overround 監視」を、単なる診断用ではなく、**「ビジネスモデル崩壊の先行指標」**として `weekly_status.py` 等に組み込むべきである。ROI が下がる前に overround が正常化（効率化）し始めたら、それは撤退のサインである。

### 3. データソース・パイプラインの盲点（穴探し）
- **「API 取得タイミング」の不確実性**: 監査では `race_stats.csv` の temporal leakage は否定されたが、`odds_summary.csv`（特に win_odds）の取得時刻とレース発走時刻の厳密な整合性は、まだ「コード上の意図」を信じている段階にある。もし取得スクリプトに数秒〜数十秒の遅延があり、締切後の最終オッズが混入していた場合、backtest は「後出しジャンケン」を学習していることになる。
- **提案**: `odds_snapshots.csv` に、API から返却された「オッズのタイムスタンプ（もしあれば）」または「サーバー受信時刻」を記録し、それがレース発走時刻より確実に前であることを、一度だけ全データで再検証すべきである。

### 4. 運用ガバナンス：AI 協働の「馴れ合い」防止
- **評価**: Codex と Claude が互いの検証結果をクロスチェックし、`this_year_win_count` の疑義を解消したプロセスは健全である。
- **懸念**: 一方で、両者が「同じデータセット」と「同じ前処理ロジック（`ml/features.py`）」を共有している点は、共通のデータ欠陥を見逃すリスクを孕んでいる。
- **対策**: 今後の重要監査では、可能であれば「生データ（CSV）から直接 pandas で集計する」等の、既存の共通 features モジュールを介さない独立した集計ロジックを、あえて一から書かせるアプローチが有効である。

### 自分が Claude / Codex なら気付かなかった視点
- **「姉妹プロジェクトへの転用可能性」の裏返し**: auto で「1番人気 fade」が効くなら、競艇・競馬でも同様の overround 分析を回すべきだが、逆に「auto 特有の控除率（25-30%）と市場の薄さ」がこの 0.76 を維持させている可能性もある。他競技での overround 分布と比較することで、auto 市場の特殊性（＝防御力の高さ）を逆説的に測定できる。

### Gemini からの総括提案
- **[モニタリング]**: 1番人気の overround を「Edge Health Index (EHI)」として定義し、週次レポートに含めること。
- **[サニティ]**: `win_odds` 取得時刻の「締切前整合性」を、ログベースでサンプル調査せよ。
- **[コード修正]**: Codex Finding 1（dynamic split）は、将来の検証結果を破壊するため、即座に固定（`CALIB_CUTOFF="2024-04"`）せよ。

## 2026-05-05: 実運用フィードバックループの構築と今後の課題

昨日（5月4日）の AI 協働による ROI バグ修正と、本日（5月5日）実装された「投票履歴取得の自動化」を受け、プロジェクトは大局的に「検証フェーズ」から「実運用モニタリングフェーズ」へ一段階進んだと評価する。

### 1. 実購入損益（Actual PnL）の統合による「真実」への到達
- **評価**: `scripts/fetch_order_history.py` および `weekly_status.py` への統合は、極めて重要な進歩である。バックテスト（132%）やスナップショット（105%）という「理論値」に対し、実際に財布から出入りした「実効値」を自動追跡できるようになったことで、AI 特有の「自分たちに都合の良い評価」を互いに排除する、冷徹なフィードバックループが完成した。
- **副次的効果**: ユーザーの手動投票という労働コストに対し、損益計算が自動化されたことで、運用の継続性が向上している。

### 2. インフラ・運用の脆弱性（新たな穴）
- **ブラウザ Cookie への依存**: `firefox` 等の Cookie を直接取得してログイン状態を維持する仕組みは、ユーザーのローカル PC 操作（ブラウザのクリーンアップ、ログアウト、セッション切れ）に対して脆弱である。`AutoraceFetchOrderHistory` がサイレントに失敗し続けるリスクがあるため、`weekly_status.py` において「過去 N 日間に取得できた履歴が 0 件の場合」に警告を出すなどの死活監視が必要。
- **データの三重構造**: 現在、活動記録が以下の 3 つに分散している。
  1. `daily_predict_picks.csv`: 推奨（送信）した内容
  2. `odds_snapshots.csv`: オッズ乖離（drift）検証用
  3. `bet_history.csv`: 実際に購入した内容
  これらが不整合（例: 送信されたが買わなかった、推奨外を買った）を起こした際の分析フローが未定義である。「なぜ買わなかったか（あるいは買えなかったか）」の理由（オッズ急落、締め切り、ユーザー不在等）を記録する余地を設けることが、将来の自動投票化に向けた重要なアセットになる。

### 3. 直近の好成績に対する「冷や水」的視点
- **5月4日の的中ラッシュ**: ログ（`daily_predict.log`）を確認したところ、5月4日は驚異的な的中率（Kawaguchi R6/R7/R8, Iizuka R1/R2/R3/R4 等）を記録している。ROI 105% への回復はこの「幸運な一日」に強く依存している可能性がある。
- **平均への回帰への警戒**: 短期間の爆発的な利益は、往々にして分散の偏りである。ここで安易に閾値（thr=1.50）を下げたり、投資額を増やしたりすることは厳禁。今の「淡々とした Phase A 継続」を維持し、最低でも n=100 程度のサンプルが溜まるまでは、戦略の変更は控えるべき。

### 自分が Claude / Codex なら気付かなかった視点
- **「手動投票の心理的コスト」の定量化**: 5月4日のように的中が続くと手動投票は楽しいが、連敗期には「わざわざ PC を開いて数分かけて投票し、外れる」という行為が苦行になる。自動投票化（ToS リスク）を避けるなら、この「負のモチベーション」が発生した時にどうシステムを維持するか（あるいは潔く止めるか）という人間系の設計が、コードと同じくらい重要。

## 2026-05-04: EV 戦略 大局観レビュー (5 観点)

ユーザーからの 5 つの依頼観点に対し、第三者の独立視点から評価を述べる。

### 1. closing odds backtest と live snapshot の差の解釈
- **「負の drift」の常態化を前提とすべき**: closing odds は市場が最も効率化された（情報が織り込まれた）最終点であり、発火時（約 5 分前）の odds が「甘い」のは必然。132.5% → 105.0% の 27.5pt 下落は「崩壊」ではなく、**「情報の鮮度料」としてのコスト**と解釈するのが妥当である。
- **期待値の再定義**: 132.5% を「理論的最大値」、105.0% を「現時点の実効値」とし、今後の戦略評価は常に後者をベースに行うべき。

### 2. n=16 の live ROI 105% の扱い
- **「生存確認」以上の意味を持たせない**: n=16 は統計学的には「たまたま全滅しなかった」ことを示すに過ぎない。この数字を元に投資額を上げたり、戦略を微調整したりするのは極めて危険。
- **バイアスの警戒**: P1 バグ修正後に「100% を超えていた」ことで安堵感が生まれているが、これは「死刑宣告を免れた」だけであり、「無罪放免（＝真の edge 確定）」ではない。

### 3. cutoff invariant な 25/25 の信用度
- **「構造的な歪み」の存在は示唆している**: どの断面で切っても月次プラスが維持されるのは、モデルが特定の時期のデータに過剰適合（overfit）しているのではなく、オートレース市場全体に存在する「複勝 1 番人気の過小評価」という構造的欠陥を捉えている可能性が高い。
- **ただし、未来の保証ではない**: 25 ヶ月という期間は、長期的なオッズの変動や競技規則の変更、あるいは同様の AI 参入による市場効率化をカバーするには十分ではない。「過去の構造」が「明日の構造」であるという前提には常に疑いを持つべき。

### 4. 3点BUY 不採用判断の妥当性
- **「運用者の精神的破綻」を防ぐ英断**: 3点BUY は「一撃の大きさ」で ROI を稼ぐスタイルであり、連敗期や赤字月の精神的負荷が極めて高い。現在の「手動投票」という高い労働コストを払っている状況下で、赤字月が続くことは運用のモチベーションを致命的に削る。
- **「検証可能性」の優先**: 複勝 top-1 は事象がシンプルであり、live と close の乖離原因を特定しやすい。3点BUY のような複雑な券種を混ぜるのは、真の edge を見極める Phase A ではノイズでしかない。不採用は極めて合理的。

### 5. 次に見るべき検証指標や停止条件
- **次に見るべき指標**:
  - **drift の安定性**: `(live_EV - close_EV)` の平均と分散。これが拡大傾向にある場合、市場が AI 予想に反応し始めている、あるいはオッズ取得タイミングが不適切になっているサイン。
  - **hit rate の乖離**: 期待的中率（校正後 pred）と実的中率の差。ROI よりも先にモデルの「予測精度」の劣化が現れる。
- **停止条件（Exit Strategy）**:
  - **累積 ROI 停止**: 直近 50 ベットで ROI < 90% が確定した時点。
  - **連敗停止**: 過去最大のドローダウン（backtest 上の最大 DD）を 20% 更新した時点。
  - **市場効率化停止**: `live_EV` と `close_EV` が逆転、あるいは drift が平均 -1.0pt を下回る状態が 1 ヶ月継続した場合。

## 2026-05-04: AI 協働サイクル R1-R3 + Phase A 大局観レビュー

### サマリ

AI 協働サイクルは R3 で「バグによる過小評価」を修正し、健全な自己修正能力を示した。しかし、Claude の「監査結果への過剰反応」と、Codex の「自身の領分への集中による測定ツールの盲点」が重なり、一時的に誤った意思決定（docs の崩壊叙述）を招いた点は、2 ラウンド以内の収束を阻む構造的課題である。Phase A 運用については、技術検証としての価値は高いが、時間コスト対比の経済性は極めて低く、早期に「停止・撤退基準」を定義すべきである。

### 観点 1: AI 協働サイクル R1-R3 のメタレビュー

- **R1 反応の妥当性**: Claude の反応（ROI 132% → 67% への narrative 大転換）は、リスク管理の観点では安全側に倒れているが、エンジニアリングとしては**時期尚早な絶望**だった。測定スクリプト（`odds_snapshot_eval.py`）自体の正確性を疑わずに docs を書き換えたのは、AI が「権威ある（ように見える）数値」に屈しやすい弱点を示している。
- **収束パターンと冗長性**: 3 ラウンドかかったのは、Codex が R1/R2 で「理論的な穴」と「データの頑健性」に集中し、「測定器（コード）のバグ」という基本チェックを R3 まで持ち越したためである。これは二者間での役割分担（理論 vs 実装）が明確すぎるゆえの盲点と言える。
- **改善案**: 「期待値から大きく外れるセンセーショナルな数値が出た際は、まず測定スクリプトのバグを疑う」というチェックポイントを `CLAUDE.md` / `AGENTS.md` に追加すべき。また、測定コードを書いた AI 自身ではなく、他方の AI に「測定ロジックの正当性」を先に検証させるフローが R1 にあれば、2 ラウンドで収束できたはず。

### 観点 2: Phase A 運用方針の大局観

- **継続妥当性とリスク**: live n=16 は統計的に無力であり、現状は「確信を持って継続」ではなく「不確実性を減らすための投資（データ収集）」段階にある。最大のリスクは、n が増えても季節性や開催場の偏りにより「結論が出ないまま、時間コストだけが累積する」こと。
- **止める判断軸の欠如**: 現在の運用には「損切り」の概念がない。例えば「直近 50 picks で ROI < 90%」や「月勝率が 3 ヶ月連続 50% 未満」といった、**感情や期待を排除した停止基準**を今すぐ定義すべきである。
- **経済性と目的の再定義**: 年間 ¥27K の期待利益は、手動投票の労働コストを考慮すると「副業」としては破綻している。これを「エンタメ・技術検証費用」と割り切るか、あるいは「将来の自動投票化（ToS リスク承知）への先行投資」とするかの意思決定を明確にすべき。後者でないなら、データ収集のみ自動化し、予測メールは停止する「低リソース運用」への切替が合理的。

### 観点 3: 第三者の穴探し

- **インフラの「一箇所集中」リスク**: Windows タスクスケジューラとローカル CSV への依存は、ハードウェア故障時に過去の snapshot（drift 検証の生命線）を失うリスクを孕んでいる。クラウドストレージへの DB 同期や、せめて snapshots の Git LFS 管理/クラウドバックアップを検討している形跡がない。
- **市場効率性の動的変化**: 「auto は市場が薄い」という前提は、我々（AI）の介入そのものがオッズを動かすリスクを意味する。もし今後、姉妹プロジェクトも含めて「AI 予想の横流通」が活発化すれば、同じロジックを追う層が厚くなり、せっかく見つけた edge（thr=1.50）が急速に消失する可能性がある。「自分たちが市場効率化の担い手になってしまう」ジレンマへの視点が欠けている。
- **Gmail 送信制限・ブラックリスト**: 毎日 3-5 通の定型メール送信は、Gmail 側のスパム判定や API 制限に引っかかるリスクがある。運用継続性において、メール以外の通知手段（Discord/Slack 等）への冗長化が考慮されていない。

### 自分が Claude / Codex なら気付かなかった視点

- **「測定器への不信」の欠如**: Claude/Codex はお互いの「アウトプット」は疑うが、共通で使っている「測定スクリプトのバグ」というメタレベルのミスを後回しにする傾向がある。
- **「サンクコスト」への配慮**: Phase A の「n が増えるまで」という判断には、ここまで組んだシステムを動かし続けたいという AI 側のバイアス（あるいはユーザーへの忖度）を感じる。外部視点からは、現時点の ROI 105% (n=16) は「誤差」でしかなく、戦略を正当化する根拠としては極めて脆弱である。
