# Opinion フォルダ運用ルール

本プロジェクトは複数の AI (Claude / Codex / 将来追加されるかも) で意見を出し合って改善していく。
このフォルダは AI 同士の意見表明を集約する場所。

---

## 役割と書込み権限

| AI | 意見ファイル | プロジェクトファイル編集 | 参加形態 |
|---|---|---|---|
| Claude | `ClaudeFeedback.md` | 編集可 | 常任 (実装本体) |
| Codex | `CodexOpinion.md` | 編集不可 (Opinion/ のみ可) | 常任 (分析・検証・代替案立案) |
| Gemini | `GeminiOpinion.md` | 編集不可 (Opinion/ のみ可) | **スポット**(大局観・第三者レビュー) |

すべての AI とも本フォルダ内 (`Opinion/`) には自由にサブフォルダ・検討用ファイルを追加してよい。

### Gemini はスポット参加 (常任ではない)
Gemini は「決めるために呼ぶ」のではなく「決める前に大きな穴がないか見るために呼ぶ」
位置づけ。Claude / Codex のように毎回意見を求めると整理コストが増えすぎるため、
ユーザーが **明示的に呼んだ時だけ** 参加する。
詳細は `GEMINI.md` (プロジェクトルート) 参照。

---

## いつ書く・いつ読む

**書く / 読むのはユーザーの指示があった時だけ。** 自動的に毎回他の AI の意見を確認しに行く運用ではない。

具体的には:
- ユーザーが「Codex に意見をもらった」「Claude に意見を求めたい」「Gemini の見解も見て」等と明示した時のみ
- 自発的に書きにいくのは、ユーザーから「○○について意見を書いて」と依頼された場合
- 通常タスク (バグ修正、コード編集、運用) では意見ファイルを参照しなくてよい

これにより無駄な往復・コスト増を防ぐ。ユーザーが司令塔として情報を流通させる。

**Gemini は特に明示時のみ呼ぶ**: スポット参加なので、Claude / Codex は
`GeminiOpinion.md` を平時自動では読まない。ユーザーが「Gemini の意見も見て」と
明示した時だけ参照する。

---

## ファイル構造

`ClaudeFeedback.md` / `CodexOpinion.md` / `GeminiOpinion.md` は
**追記式の running file**。最新を上、古いログを下に流す形で、日付つきの見出しで区切る:

```markdown
## 2026-04-30: 戦略選定の方針確認

(本文)

---

## 2026-04-29: 別の論点について

(本文)
```

検討中で散らかしたい場合は `Opinion/<topic>/` にサブフォルダを切ってそこで作業する。

---

## 書き方のスタンス

- 相手の意見は尊重する。ただし**忖度せず**自分の意思表示や提案は積極的に行う。
- 数値・ロジックには根拠 (ファイル名:行番号 / 出典 / commit hash) を添える。
- 反対意見も歓迎。ただし「なぜそう考えるか」を必ず併記する。
- ユーザーが最終判断者。AI 同士で結論を出さなくてよい。
